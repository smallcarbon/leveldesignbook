优化

诊断关卡的性能问题并对其进行调整以使其运行更顺畅

当您尝试玩自己的关卡时，游戏运行是否非常缓慢？您的玩家或试玩测试人员是否抱怨延迟、帧速率低或加载时间长？如果是这样，那么也许是时候优化你的关卡以使其运行更高​​效了。

什么是优化？

优化 是诊断问题并简化游戏或关卡性能的过程。它往往非常复杂和具体，远远超出了这本The Level Design Book 中文版的范围。就像映射一样，更好地优化的唯一真正方法是优化大量游戏，并从您的经验中学习。

谁的工作？

在团队中工作时，许多性能考虑因素将远远超出关卡设计师的职责范围。

例如，如果寻找两个 NPC 使游戏帧率降至 5 FPS，那么优化就是引擎/AI 程序员的工作。然而，如果关卡设计师试图创造一场与数十个 NPC 的大规模战斗，而游戏本来就不应该支持这种类型的遭遇战，那么显然关卡设计师的错误可能更大。

大型商业工作室聘请技术艺术家和技术关卡设计师，他们的主要职责可能涉及优化艺术资产和水平。但在较小的开发团队中，优化基本上是每个人的责任。

何时优化

“不算太早，也不算太晚。”

过早优化是 “过早优化” 这是浪费时间，因为你的项目将会发生巨大的变化。为什么要优化玩家可能永远不会看到的东西？

但你怎么知道最终版本中是否会有某些内容呢？可能只有当它运行良好时......有时你只有在尝试优化它之后才能做出判断。

规划优化

理想情况下，优化是您从一开始就计划好的事情。早在 前期设计，您应该能够回答有关项目技术范围的以下问题：

  • 目标平台是什么？ 例如，为手机制作开放世界游戏比为桌面游戏设备制作复杂得多。

  • 水平有多大？ 大型开放世界城市或 RPG 大陆需要以较小的块建造，并在游戏过程中小心地加载/卸载，并且 白盒 因此。

  • 水平有多密集？ 现实的户外世界可能需要大量昂贵的重叠细节，而风格化的手绘世界可能更加稀疏和简化。

  • 开放程度如何？ 具有频繁门的狭窄室内关卡可以更好地利用遮挡剔除，而大型开放的室外景观则难以划分为多个部分。

  • 什么是摄像头系统？ 例如，第一人称相机可以看到长视线，因此密集的关卡可能需要某种遮挡剔除。第三人称固定摄像机角度意味着您可以对视锥体剔除进行强大的控制，如果特定摄像机角度滞后，那么您可以简单地重新定位摄像机或将摄像机转开。

绩效预算

视频游戏在资源有限的计算机上运行。当您的项目过度使用其时，就会出现滞后的低帧率 预算 对于以下一种或多种资源类型：

  • 游戏由加载到的文件组成 记忆。当您启动游戏时，它将核心文件移动到称为 RAM 的更快类型的短期内存中。如果有太多文件需要加载到 RAM 中，那么加载时间将会很长。

    • 如果您的关卡使用大量不同的详细模型、纹理、声音或动画，那么它可能会超出其范围 内存预算。

  • 这 中央处理器 是计算机的主要部分，运行文件系统、控件、游戏逻辑、人工智能和音频。如果你有太多的 NPC 在一座复杂的大城市中航行，那么 CPU 可能必须努力工作才能模拟他们所有的 AI。或者想象一下，虚拟城市中有数千只昆虫，并且您每秒都在不断地创建/删除昆虫。

    • 如果您的关卡具有许多不同的模拟对象，那么它可能是 CPU 限制。

  • 这 图形处理器 （图形处理单元）负责在屏幕上渲染图像。它必须收集所有可见的 2D 精灵和 3D 模型，并使用称为着色器的特殊 GPU 程序对其进行处理。当需要绘制的对象太多或着色器非常复杂时，GPU 会花费更长的时间来渲染相机视图。

    • 如果您的关卡具有视觉密集的对象，甚至数千个简单对象，那么它可能是 GPU 限制。

在谈论预算时，游戏开发者扩展了这个财务比喻：

  • 贵 = 消耗大量内存或处理时间

    • 例如：“哇，那个着色器让我们的游戏以 10 FPS 运行，太贵了！”

  • 便宜 = 使用很少的内存或处理时间

    • 示例：“当玩家距离 10 米时，我们更换为廉价的水着色器。”

分析

游戏运行缓慢的原因有多种：内存、CPU、GPU 等。要找出游戏的哪个部分导致速度缓慢，我们必须测量游戏引擎正在执行的操作 - 这称为 剖析.

永远不要尝试根据信念或直觉进行优化，因为您可能错误地认为导致游戏运行缓慢的原因是错误的。 分析应该是任何优化过程的第一步！ 幸运的是，现代游戏引擎具有一套强大且易于使用的分析工具。

如何在 Unity 中分析关卡

  • 试玩测试与 游戏选项卡中的统计面板 寻找基本方向

  • 如果您受 CPU 限制，请使用 分析器 并查找哪些代码函数运行频率很高或似乎很耗时；如果您不是程序员，您应该与程序员交谈

  • 如果您受 GPU 限制，请打开 GPU 分析器 甚至是 帧调试器，并尝试找出是否在错误的时间或地点渲染了太多内容，或者剔除/批处理/实例化是否在某个地方发生了破坏

如何在虚幻中分析关卡

  • 在 Unreal 中，转到项目设置 → 常规设置 → 帧速率和 禁用平滑帧速率， 它将帧速率调整为一定的增量并会误导您

  • 在试玩测试时，在控制台中输入 统计单位图 显示基本 CPU 和 GPU 时间

  • 如果您受 CPU 限制，您可能需要使用 虚幻见解，也许你有太多的演员或者你的一些蓝图调用很昂贵

  • 如果您受 GPU 限制，请在控制台中输入 统计场景渲染 并查看您的绘制调用和帧率的走向

绘制调用

除非有很多复杂的游戏系统或脚本发生，否则大多数游戏关卡通常都是 GPU 限制： 显卡有太多工作要做。

在分析 GPU 性能时，需要关注的主要统计数据是 绘制调用， 发送到 GPU 的渲染命令。更多的绘制调用通常会更糟。你希望这个数字更小。

为了减少绘制调用，引擎实现了某种形式的（a） 批处理 或 (b) 实例化。

配料 是指 CPU 将多个绘制调用合并为一个绘制调用。

想象一个关卡有 10 棵不同的树；我们可以将它们全部组合成一棵巨型树，并将其作为一次绘制调用发送到 GPU。这种方法通常需要一些预先计算和精心构建，但它适用于多个不同的网格和对象。

参见统一： 静态配料，虚幻： 层次结构的详细程度，戈多： 配料

实例化 是指 GPU 多次重复相同的绘制调用。

想象一个关卡有 10 棵相同的树；我们可以告诉 GPU 对每棵树重复相同的渲染工作。此方法几乎不需要关卡设计工作或准备工作，但它仅适用于同一网格的精确副本。非常适合 模块化 工作流程，它使用相同模块化套件图块的许多副本。

参见统一： GPU实例化，虚幻： 分层实例静态网格体

在游戏开发中，“实例”也意味着任何游戏对象或数据的“复制”或“克隆”。不要混淆实例与实例的一般用法。 GPU 实例化。

灯光

前向渲染 表示显卡（GPU）直接处理每个对象前进 方式，分别渲染和照明每个 3D 对象。这里通常应该避免重叠的灯光，并使用尽可能少的光源。

延迟渲染器 延迟照明计算，直到将所有对象收集在一起，然后立即照亮所有可见像素。在这里，您可以使用许多重叠的灯光和光源，但透明物体可能看起来很糟糕。

引擎通常同时使用这两种技术，或者混合搭配不同的部件。

要审查...

优化游戏是一个复杂的过程，涉及各个学科和开发人员。这远远超出了本书的范围。尽管如此，我们还是尝试给出在 Unity 和 Unreal 等常见引擎中优化 3D 关卡的基本思路：

  1. 看性能 统计/分析器 看看什么是慢的， 不要只是猜测

  2. 如果是 CPU 限制， 也许有太多活动对象或 NPC，或者关卡脚本行为不当，或者关卡太大

    • 减少活动物体或 NPC 的数量； 重新设计遭遇战

    • 调试你的关卡脚本； 向别人寻求帮助

    • 这也可能是一般游戏代码问题，与您无关

  3. 如果是 GPU 限制， 也许有太多的灯光或环境美术对象，或者材质/着色器太复杂，或者关卡太大

    • 减少灯光，使用更简单的照明模式，关闭阴影投射

    • 使用相同的材质组合多个对象 批处理

    • 使用相同的材质和网格组合多个对象 GPU实例化

    • 考虑使用某种 遮挡剔除 / 可视性系统

    • 简化材料， 使用更少的纹理采样器和渲染命令

    • 这也可能是一般游戏代码问题，与您无关
